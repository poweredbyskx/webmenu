- UptimeRobot настроен на `https://kakao.com.tm`, плюс включена отдельная проверка **SSL Certificate Expiry Reminder** (за 30 дней до истечения)

### ⚠️ SSL-сертификат истекает 2026-07-21 — автопродление сломано, отложено
- Сертификат: `kakao.com.tm`, истекает **2026-07-21**. Автопродление (`certbot renew`) не работает:
  - HTTP-01 проверка (порт 80) падает с "Connection refused" при "secondary validation" от Let's Encrypt — стабильно повторяется, похоже на избирательную блокировку в туркменской сети (порт 80 при этом реально открыт и работает — проверено внешне)
  - DNS-01 проверка тоже не получилась с первой попытки (TXT-запись не была подтверждена, `NXDOMAIN`) — DNS домена управляется `ns3/ns4.telecom.tm` (гостелеком, без публичного API)
- Решили отложить, но **вернуться нужно до 2026-07-21** — UptimeRobot теперь тоже напомнит
- Варианты на будущее: (а) повторить DNS-01 вручную через панель DNS, если она реально даёт добавлять TXT-записи; (б) перенести DNS на Cloudflare — тогда SSL продлевать не нужно вообще (Cloudflare делает это сам); (в) купить платный сертификат на 1-2 года с проверкой по email вместо HTTP/DNS

### ~~Проверить SECURE_SSL_REDIRECT~~ ✅ выяснено — ничего делать не нужно
- Nginx уже редиректит `http://` → `https://` (`return 301 https://kakao.com.tm$request_uri;`), проверено конфигом и внешним запросом — трогать `SECURE_SSL_REDIRECT` в Django не нужно

### Sentry (отслеживание ошибок)
- Ловит ошибки, которые НЕ кладут весь сайт (например, у одного пользователя что-то не сработало) — без этого о них никогда не узнать
- Бесплатный тариф достаточен для такого проекта

### Безопасный процесс деплоя
- Сейчас деплой — руками через SSH (`git pull` + `systemctl restart`), без проверки, что новый код реально запустился (вчерашний баг с `django-ratelimit` от этого и не поймали сразу)
- Минимум: простой bash-скрипт на сервере, который после рестарта сам делает `curl` и явно говорит "OK"/"FAIL"
- Деплоить только когда GitHub Actions (CI) зелёный
